Topic 03

Domain Name System

The internet's phonebook — how human-readable domain names are translated into machine-readable IP addresses every time you visit a website.

Section 01

What is DNS?

Definition

The Domain Name System (DNS) is a hierarchical, distributed naming system that translates human-readable domain names (like www.google.com) into numerical IP addresses (like 142.250.190.46) that computers use to identify each other on a network.

Without DNS, you would have to memorise a different number for every website you visit. DNS acts as the internet's phonebook — you look up a name, it gives you the address. It was developed in 1983 by Paul Mockapetris and remains one of the most critical components of the internet today.

📖 Phonebook of the Internet

Maps easy-to-remember domain names to numeric IP addresses automatically.

🌐 Distributed System

No single server holds all records — thousands of servers work together worldwide.

⚡ Fast Resolution

DNS queries are resolved in milliseconds using caching and hierarchy.

🔁 Automatic

Happens invisibly every time you type a URL — you never have to do anything manually.

🗂️ Hierarchical

Organised in a tree structure: Root → TLD → Domain → Subdomain.

💾 Cached

Results stored temporarily to speed up repeated lookups and reduce server load.

Flowchart

How DNS Resolution Works

Step-by-step DNS Lookup — what happens when you type a URL

Every time you visit a website, this entire process happens automatically in the background — usually in under 100 milliseconds.

🧑‍💻 User types www.google.com in browser
Step 1 — Browser Cache Check
Browser checks its own DNS cache. If the IP was looked up recently, it uses the cached result immediately.
Cache hit?
Step 2 — OS / Resolver Cache
If not in browser cache, OS checks its own DNS resolver cache from previous lookups.
Step 3 — Recursive Resolver
Query sent to ISP's Recursive DNS Resolver. It does the heavy lifting of finding the answer.
Step 4 — Root Name Server
Resolver asks a Root Server. There are 13 root server clusters worldwide. Root server points to correct TLD server.
Step 5 — TLD Name Server
TLD server (e.g., .com) points to the Authoritative Name Server for google.com.
Step 6 — Authoritative Name Server
Holds the actual DNS records for google.com and returns the IP address.
Step 7 — Response & Cache
IP address returned to browser. Result cached based on TTL (Time to Live) value.
✅ Browser connects to IP and loads the website
Section 02

DNS Hierarchy

The Tree Structure of DNS

DNS is organised like an inverted tree. At the very top is the Root, below that are Top-Level Domains (TLDs), then Second-Level Domains, and finally Subdomains. Each level is separated by a dot in the domain name.

Example: mail.google.com

. (Root — invisible, at the end of every domain)
🌍 .com — Top-Level Domain (TLD)
🏢 google — Second-Level Domain (SLD)
📧 mail — Subdomain

Types of Top-Level Domains

Generic TLDs (gTLDs)

.com (commercial), .org (organisation), .net (network), .edu (education), .gov (government)

Country Code TLDs (ccTLDs)

.in (India), .uk (United Kingdom), .us (USA), .jp (Japan), .au (Australia)

New Generic TLDs

.app, .dev, .tech, .io, .ai — newer extensions introduced after 2012

Section 03

Types of DNS Servers

The four servers involved in every DNS lookup
🔄 Recursive Resolver

First stop after your device. Acts on your behalf querying other servers. Usually run by your ISP or a public DNS provider like Google 8.8.8.8 or Cloudflare 1.1.1.1.

🌍 Root Name Server

Highest level of DNS. There are 13 root server addresses (A through M) managed by ICANN, NASA, and Verisign. They direct queries to the correct TLD server.

🗂️ TLD Name Server

Handles queries for a specific TLD like .com or .in. Maintained by domain registries. Points the resolver to the authoritative server for the specific domain.

✅ Authoritative Name Server

Final authority for a domain. Holds the actual DNS records and gives the definitive IP address answer. Managed by the domain owner.

💾 DNS Cache

Stores recent DNS results at every level — browser, OS, resolver — to avoid repeating the full lookup process each time.

Section 04

DNS Record Types

Common DNS records stored in the Authoritative Name Server

Each record type stores different information about a domain. These are the most important types you need to know:

A Address Record

Maps a domain name to an IPv4 address. The most common and fundamental record type.

google.com → 142.250.190.46
AAAA IPv6 Address Record

Maps a domain name to an IPv6 address — the newer 128-bit IP standard.

google.com → 2404:6800:4009::200e
CNAME Canonical Name

Alias — maps one domain name to another. Used to point subdomains to the main domain.

www.example.com → example.com
MX Mail Exchange

Specifies the mail server responsible for accepting email for the domain.

gmail.com → smtp.google.com
NS Name Server

Identifies which DNS servers are the authoritative name servers for the domain.

example.com → ns1.provider.com
TXT Text Record

Stores arbitrary text — used for domain verification, SPF spam prevention, and security.

v=spf1 include:_spf.google.com ~all
PTR Pointer Record

Reverse DNS — maps an IP address back to a domain name. Used in email security.

142.250.190.46 → google.com
TTL Time to Live

Value in every record — how long in seconds the result can be cached before re-querying.

TTL: 3600 = cache for 1 hour

Section 06

Advantages & Limitations

  • Human-friendly — easy domain names instead of IP numbers
  • Highly scalable — handles billions of queries daily
  • Fast — caching ensures most lookups are near-instant
  • Flexible — IP can change without users noticing
  • Decentralised — no single point of failure
  • Supports load balancing via multiple A records
  • DNS poisoning — attackers can forge DNS responses
  • DDoS attacks on DNS servers can take down many sites
  • Propagation delay — DNS changes take up to 48 hours to spread
  • Privacy concern — queries unencrypted by default (solved by DNS over HTTPS)
  • Caching issues — outdated records can cause incorrect routing
